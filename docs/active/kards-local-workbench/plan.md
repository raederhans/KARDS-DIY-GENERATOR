# KARDS 本地工作台、导出诊断与参考匹配实施计划

## 目标

在现有 React + TypeScript + Canvas 静态架构内，把右栏整理为外观、卡库、导出、参考四个标签，并补齐本地卡库 CRUD、结构化导出诊断、参考筛选排序和安全的自动卡图。

## 约束

- 不新增依赖，不修改 README，不引入网络图库、批量导出、回收站或游戏规则。
- 左侧字段编辑和中间 Canvas 预览保持现有职责。
- 卡库保持 `version = 1`、单 JSON 文件、最多 200 条、最多 2 MB，并继续剥离用户上传图片。
- 自动卡图默认开启，但上传、导入、显式应用、载入整卡和裁剪等明确用户行为优先。
- 导出诊断只存在于当前 UI 运行期，不写入卡牌、项目、草稿或卡库。
- 测试优先扩展现有测试文件；生产行为遵循先失败、后最小实现、再重构的循环。

## 阶段

1. 建立四标签工作台外壳，保留现有行为和固定底部区。
2. 为参考目录补齐元数据，实现 AND 筛选、稳定排序、唯一自动匹配和 artwork 来源保护。
3. 为 v1 本地卡库实现规范化、创建、更新、删除和只读/损坏边界。
4. 为导出新增结构化 preflight、一次渲染结果和保存目标语义。
5. 接通 App 运行时状态、偏好存储、可访问性和响应式布局。
6. 完成目标测试、类型检查、全量验证、Pages 构建、依赖审计和有限浏览器 smoke。
7. 三名独立复核者分别审查代码、架构和测试；修复问题后整合、推送、归档并清理 worktree。

## 验收

- `npm run validate` 通过。
- Pages 模式生产构建通过，依赖审计无已知漏洞。
- 快速切换参考时旧请求无法覆盖新卡，且加载期间的无关字段编辑保留。
- 用户卡图永不被自动匹配覆盖。
- 卡库并发写入基于锁内重读，不会用 UI 旧快照覆盖磁盘；损坏文件和只读环境不允许写。
- 浏览器下载显示“已触发”，目录保存只有写入完成后才显示“已写入”。
- 四标签支持键盘与 ARIA tab 语义，窄屏工作台位于预览下方。
